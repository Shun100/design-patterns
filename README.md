# design-patterns
デザインパターン学習用

## 実行方法
`npm start hoge/hoge.ts`

## Summary
- `01_TemplateMethod`
  - 重要度: ★☆☆
  - 概要: 親クラスで処理の枠組みを決め、子クラスで枠組みの具体的な内容を定めるパターン
  - コメント: 普通に継承を使っていればいつの間にかできていることなので、デザインパターンとして覚える必要はない
- `02_Singleton`
  - 重要度: ★☆☆
  - 概要: クラスが1つのインスタンスのみを持つことを保証し、このインスタンスへアクセスするためのグローバルな方法を提供するパターン
  - コメント: マシンリソースに余裕がある現代ではあまり使わない むしろ同じインスタンスを共有すると思わぬバグを生むので下手に使わない方が良い
- `03_Adapter`
  - 重要度: ★☆☆
  - 概要: あるクラスのインタフェースを、そのクラスを利用する側が求める他のインタフェースへ変換するパターン
  - コメント: 特に意識しなくてもできていることなので、デザインパターンとして覚える必要はない
- `04_Iterator`
  - 重要度: ★★☆
  - 概要: コレクションの内部構造を利用者に見せずに、その要素に順番にアクセスする方法を提供するパターン
  - コメント: 自分で実装する機会はあまりないが、ライブラリ等ではよく使われているので理解はしておいた方が良い
- `05_Factory`
  - 重要度: ★★★
  - 概要: 親クラスでインスタンスの生成方法を定め、具体的に何をどうやって作るかは子クラスで定めるパターン
  - コメント: よく使う
- `06_Facade`
  - 重要度: ★☆☆
  - 概要: 複雑な内部処理をまとめ、システムの外部に簡素化されたインタフェースを提供するパターン
  - コメント: 特に意識しなくてもできていることなので、デザインパターンとして覚える必要はない
- `07_Prototype`
  - 重要度: ★☆☆
  - 概要: 原型となるインスタンスをコピーして新しいインスタンスを生成するパターン
  - コメント: 特に意識しなくてもできていることなので、デザインパターンとして覚える必要はない
    - Shallow Copy, Deep Copyの理解は必須
- `08_Builder`
  - 重要度: ★★★
  - 概要: 同じ生成手順で異なる材料を使って異なるオブジェクトを生成するパターン
  - コメント: よく使う
- `09_AbstractFactory`
  - 重要度: ★☆☆
  - 概要: 関連したオブジェクト（部品）のセットを、具体的なクラスを指定することなく生成するためのインタフェース（API）を提供するパターン
  - コメント: Factoryパターンを抽象化したものだが、ここまで行くと過剰な設計になることが多い
- `10_Strategy`
  - 重要度: ★☆☆
  - 概要: 複数のアルゴリズムを定義し、切り替えられるようにするパターン
  - コメント: 特に意識しなくてもできていることなので、デザインパターンとして覚える必要はない
- `11_State`
  - 重要度: ★★★
  - 概要: 複数の状態を別個のクラスで定義し、状態が変化したときに振る舞いを切り替えられるようにするパターン
  - コメント: よく使う 「状態」をオブジェクトと捉える考え方は非常に大切
- `12_ChainOfResponsibility`
  - 概要: クライアントからのリクエストを処理するオブジェクトを鎖のように繋げ、処理が可能なオブジェクトにリクエストを順に渡すパターン
- `13_Composite`
  - 重要度: ★★☆
  - 概要: ツリー構造を持つデータに再帰的な処理を行うパターン
  - コメント: ディレクトリツリーなどツリー構造のデータを扱うときに使う
- `14_Decorator`
  - 重要度: ★★★
  - 概要: 基本となるオブジェクトに対して柔軟に機能を追加するパターン
  - コメント: よく使うが、構造が難しいので何回か復習しないと理解できない
- `15_Proxy`
  - 重要度: ★★☆
  - 概要: 代理となるオブジェクトを通じて、間接的に目的のオブジェクトにアクセスさせるためのパターン
  - コメント: 条件分岐ではなくクラスを追加して対応する考え方は大切 ただ、Proxyパターン自体はそこまで使わないかも
- `16_Bridge`
  - 重要度: ★☆☆
  - 概要: 機能を提供するクラスと、実装を提供するクラスを独立させるパターン
  - コメント: 異なる複数のOS向けにアプリをリリースするとかでない限り使う機会は無さそう
- `17_FlyWeight`
  - 重要度: ★☆☆
  - 概要: インスタンス化されたオブジェクトを効率よく共有することで、生成されるオブジェクトやリソースの消費を抑えるパターン
  - コメント: マシンスペックが上がった現代ではあまり気にすることは無さそう
- `18_Observer`
  - 重要度: ★★★
  - 概要: 観察対象のオブジェクトに状態変化が発生した際に、そのオブジェクトの観測者に対して通知を行うパターン
  - コメント: 非常によく使う 特にフロントで「ボタンを押すとここの表示が切り替わる」ような処理はOberverパターンを使うとコードの見通しが良くなる
- `19_Mediator`
  - 重要度: ★★☆
  - 概要: 関連し合うオブジェクト間のやりとりを仲介者となるオブジェクトに集約し、オブジェクト同士が直接やり取りすることを制限するパターン
  - コメント: 仲介者を挟む設計自体は実務でもありそうだが、ゴッドクラス化するリスクがあるので注意
- `20_Memento`
  - 重要度: ★★★
  - 概要: オブジェクトの現在の状態を保存しておき、状態が変化した後でも戻せるようにするパターン
  - コメント: Undo, Redo機能を実装するなら必須
- `21_Visitor`
  - 重要度: ★★☆
  - 概要: データ構造とそれに対する操作でクラスを分けるパターン
  - コメント: Compositeパターン同様にツリー構造を持つデータ向け こちらは「データに対する操作」に主眼を置いている（Compositeパターンはデータそのものの実装も行う）
- `22_Command`
  - 重要度: ★★☆
  - 概要: 命令をメソッドではなく、独立したクラスで表現するパターン
  - コメント: 複数の命令をキューに貯めてあとで遅延実行したいケースのときに有用
- `23_Interpreter`
  - 重要度: ★☆☆
  - 概要: 構文を解析し、その結果を利用して処理を行うパターン
  - コメント: 構文解析を自分で実装する機会があまり無いので存在を知っておく程度で良い
