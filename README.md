# design-patterns
デザインパターン学習用

## 実行方法
`npm start hoge/hoge.ts`

## Summary
- `01_TemplateMethod`
  - 重要度: ★☆☆ 難易度: ★☆☆
  - 概要: 親クラスで処理の枠組みを決め、子クラスで枠組みの具体的な内容を定めるパターン
  - コメント: 普通に継承を使っていればいつの間にかできていることなので、デザインパターンとして覚える必要はない
- `02_Singleton`
  - 重要度: ★☆☆ 難易度: ★☆☆
  - 概要: クラスが1つのインスタンスのみを持つことを保証し、このインスタンスへアクセスするためのグローバルな方法を提供するパターン
  - コメント: マシンリソースに余裕がある現代ではあまり使わない むしろ同じインスタンスを共有すると思わぬバグを生むので下手に使わない方が良い
- `03_Adapter`
  - 重要度: ★☆☆ 難易度: ★☆☆
  - 概要: あるクラスのインタフェースを、そのクラスを利用する側が求める他のインタフェースへ変換するパターン
  - コメント: 特に意識しなくてもできていることなので、デザインパターンとして覚える必要はない
- `04_Iterator`
  - 重要度: ★★☆ 難易度: ★★☆
  - 概要: コレクションの内部構造を利用者に見せずに、その要素に順番にアクセスする方法を提供するパターン
  - コメント: 自分で実装する機会はあまりないが、ライブラリ等ではよく使われているので理解はしておいた方が良い
- `05_Factory`
  - 重要度: ★★★ 難易度: ★☆☆
  - 概要: 親クラスでインスタンスの生成方法を定め、具体的に何をどうやって作るかは子クラスで定めるパターン
  - コメント: よく使う
- `06_Facade`
  - 重要度: ★☆☆ 難易度: ★☆☆
  - 概要: 複雑な内部処理をまとめ、システムの外部に簡素化されたインタフェースを提供するパターン
  - コメント: 特に意識しなくてもできていることなので、デザインパターンとして覚える必要はない
- `07_Prototype`
  - 重要度: ★☆☆ 難易度: ★☆☆
  - 概要: 原型となるインスタンスをコピーして新しいインスタンスを生成するパターン
  - コメント: 特に意識しなくてもできていることなので、デザインパターンとして覚える必要はない
    - Shallow Copy, Deep Copyの理解は必須
- `08_Builder`
  - 重要度: ★★★ 難易度: ★☆☆
  - 概要: 同じ生成手順で異なる材料を使って異なるオブジェクトを生成するパターン
  - コメント: よく使う
